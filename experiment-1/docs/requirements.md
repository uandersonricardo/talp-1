# Requirements

This document is the single source of truth for the exam management system. All agents should consult it before implementing any feature.

---

## Overview

A web application for managing multiple-choice exam questions, composing exams, generating printable PDFs with randomized question/alternative order, and grading submitted responses.

---

## Data Model

### Question
- `id`: unique identifier
- `statement`: text of the question
- `alternatives[]`:
  - `description`: text of the alternative
  - `correct`: boolean — whether this alternative is a correct answer

A question may have multiple correct alternatives.

### Exam
- `id`: unique identifier
- `title`: name of the exam
- `course`: course name (used in PDF header)
- `professor`: professor name (used in PDF header)
- `date`: exam date (used in PDF header)
- `questions[]`: ordered list of question IDs selected for the exam
- `identifierMode`: `"letters"` | `"powers"` — how alternatives are labeled/scored

#### Identifier Modes
- **letters**: alternatives labeled A, B, C, D, ... — student writes the selected letter(s); answer key stores selected letters (e.g., `"B"`, `"AC"`)
- **powers**: alternatives labeled 1, 2, 4, 8, 16, 32, ... — student writes the sum of selected values; answer key stores the expected sum (e.g., `5` = alternatives 1+4)

### Individual Exam (generated artifact)
- `examId`: references parent Exam
- `sequenceNumber`: integer assigned at generation time (1, 2, 3, ...)
- `questionOrder[]`: shuffled question IDs
- `alternativeOrders{}`: per-question shuffled alternative indices

## Features

### 1. Question Management

CRUD operations for questions.

- **Create**: provide statement and a list of alternatives (each with description and correct flag). At least two alternatives required; at least one must be correct.
- **Edit**: update statement or any alternative. Changes do not retroactively affect already-generated PDFs.
- **Delete**: only allowed if question is not used in any exam.
- **List**: paginated list with search by statement text.

### 2. Exam Management

CRUD operations for exams.

- **Create**: provide title, course, professor, date, identifier mode, and select questions from existing questions. At least one question required.
- **Edit**: update any field or question set. Editing does not affect already-generated individual exams.
- **Delete**: only allowed if no individual exams have been generated for it.
- **List**: paginated list with search by title.

### 3. PDF and Answer Key Generation

Triggered per exam; user specifies the number of individual exams to generate.

**Individual exam PDF layout:**
- Header per page: course, professor, date, exam title
- Footer per page: individual exam sequence number
- Questions listed in a randomized order unique to each individual exam
- Alternatives listed in a randomized order unique to each question per individual exam
- Alternatives labeled according to identifier mode (letters or powers)
- After each question: blank space for the student to write their answer
  - Letters mode: labeled "Answer: ___"
  - Powers mode: labeled "Sum: ___"
- Final section (last page): fields for student name and CPF

**Answer key CSV:**
- One row per individual exam
- Columns: `exam_number`, then one column per question in canonical question order (same order as the exam's question list, not the shuffled order)
- Each cell contains the answer key value for that question in that individual exam:
  - Letters mode: letters of correct alternatives in the shuffled position (e.g., `"B"` or `"AC"`)
  - Powers mode: sum of powers assigned to correct alternatives in the shuffled position

**Behavior:**
- Generation is idempotent per batch: re-generating creates a new batch with new sequence numbers starting after the last generated
- Both the PDF and the CSV are available for download immediately after generation

### 4. Grading and Class Report

**Inputs:**
- Answer key CSV (generated by the system)
- Student responses CSV — expected columns: `exam_number`, then one column per question in the same canonical order as the answer key
  - Letters mode cell: string of selected letters (e.g., `"B"`, `"AC"`, or empty)
  - Powers mode cell: numeric sum (or empty)
- Grading mode: `"strict"` | `"lenient"`

**Grading logic:**
- Each question is worth 1 point (total score = number of questions unless normalized)
- **Strict**: student answer must exactly match the key; any deviation (wrong or missing alternative) → 0 for that question
- **Lenient**: score for the question = (number of alternatives correctly selected or correctly not-selected) / (total number of alternatives)

**Class report output (CSV):**
- One row per student response
- Columns: `exam_number`, `student_name` (if present in responses CSV), `cpf` (if present), per-question score, `total_score`

**Validation:**
- Warn (but do not fail) if an `exam_number` in the responses CSV has no matching entry in the answer key CSV
- Skip rows with no `exam_number`

## Non-Functional Requirements

- **Maintainability**: code must be clean, well-structured, and low-complexity. Prefer simple, explicit code over clever abstractions.
- **Performance**: the backend must handle PDF generation for up to 200 individual exams without timeout. The frontend must remain responsive during generation (use async status polling or streaming progress).
- **Usability**: UI must be clean, modern, and effective. Prioritize clear navigation, inline validation, and meaningful error messages.
- **Security**: not a priority at this stage. No authentication or authorization required.

## API Conventions

- Base path: `/api`
- Format: JSON for all request/response bodies
- Errors: `{ "error": "<message>" }` with appropriate HTTP status code
- IDs: strings (UUID v4)
- Dates: ISO 8601 strings

### Endpoints

#### Questions
| Method | Path | Description |
|---|---|---|
| GET | `/api/questions` | List questions (supports `?search=`, `?page=`, `?limit=`) |
| POST | `/api/questions` | Create question |
| GET | `/api/questions/:id` | Get question by ID |
| PUT | `/api/questions/:id` | Update question |
| DELETE | `/api/questions/:id` | Delete question |

#### Exams
| Method | Path | Description |
|---|---|---|
| GET | `/api/exams` | List exams |
| POST | `/api/exams` | Create exam |
| GET | `/api/exams/:id` | Get exam by ID |
| PUT | `/api/exams/:id` | Update exam |
| DELETE | `/api/exams/:id` | Delete exam |

#### Generation
| Method | Path | Description |
|---|---|---|
| POST | `/api/exams/:id/generate` | Generate individual exams; body: `{ "count": number }` |
| GET | `/api/exams/:id/batches` | List generation batches for an exam |
| GET | `/api/batches/:batchId/pdf` | Download generated PDF |
| GET | `/api/batches/:batchId/answers.csv` | Download answer key CSV |

#### Grading
| Method | Path | Description |
|---|---|---|
| POST | `/api/grade` | Grade exams; multipart form: `answers` (CSV file), `responses` (CSV file), `mode` (`strict`\|`lenient`) |

## UI Structure

```
/                     → redirect to /questions
/questions            → question list + create button
/questions/new        → create question form
/questions/:id/edit   → edit question form
/exams                → exam list + create button
/exams/new            → create exam form (select questions, set mode)
/exams/:id/edit       → edit exam form
/exams/:id            → exam detail: info, generation history, generate button
/grade                → grading tool: upload CSVs, select mode, download report
```
